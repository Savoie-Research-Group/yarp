======================================================================
BACKGROUND
======================================================================

The existing `condor_q` branch added a generic CondorJobManager suitable for shared-filesystem Condor installations.

That branch can:

- generate `.submit` files
- request CPUs, memory, and disk
- submit with `condor_submit`
- parse cluster IDs
- inspect jobs with `condor_q`
- manage normal YARP calculators

However, it assumes:

- executable paths are visible on the worker
- scratch paths are shared between submit and execute nodes
- SIF containers are visible at the same absolute path
- generated worker scripts can `cd` into submit-host directories
- `apptainer run` can be called from inside the worker job

Those assumptions do not hold on OSG.

OSG workers instead use isolated scratch directories and HTCondor file/container staging.

A real failed example showed:

- the job matched and started correctly
- 8 CPUs and 8 GB RAM were allocated
- the executable and input files were transferred
- the worker scratch directory was something like `/srv/scratch`
- `/home/thomas.burton/...` did not exist on the worker
- generated scripts failed because they used absolute submit-host paths
- transferring a SIF manually worked, but nested `apptainer run` was the wrong long-term design
- returning explicitly named files that were never created caused Condor output-transfer holds

So this is not fundamentally a Condor syntax problem. It is an OSG orchestration and data-staging problem.

======================================================================
OSG EXECUTION MODEL TO TARGET
======================================================================

Use this architecture:

    yarp-init
        ↓
    YARP pickle + STATUS.json
        ↓
    yarp-osg controller on the access point
        ↓
    discover ready YARP tasks
        ↓
    generate task-local input directories
        ↓
    group compatible tasks into HTCondor clusters
        ↓
    use OSDF-hosted container images
        ↓
    run inside worker-local scratch
        ↓
    return selected outputs
        ↓
    harvest and scrape outputs
        ↓
    update YARP pickle and STATUS.json
        ↓
    repeat until complete

Important OSG conventions:

1. Do not assume `/home/...` paths exist on workers.
2. Do not run nested `apptainer run` as the normal execution model.
3. Prefer HTCondor `container_image = osdf://...`.
4. Put reusable SIF files in versioned OSDF locations.
5. Use worker-local paths such as:
       ${_CONDOR_SCRATCH_DIR:-$PWD}
6. Transfer only small task-specific inputs and outputs through HTCondor.
7. Do not transfer a 700 MB–4.2 GB SIF separately for every job.
8. Prefer batched Condor clusters using `queue ... from manifest`.
9. Do not create and submit thousands of independent `.submit` files if one cluster can represent them.
10. Return only essential outputs.
11. Large reusable inputs and containers belong in OSDF.
12. Large output archives may also need OSDF rather than normal access-point return.

The user’s OSDF namespace is expected to resemble:

    /ospool/ap40/data/thomas.burton/

with URL form:

    osdf:///ospool/ap40/data/thomas.burton/

Use versioned immutable container paths, for example:

    osdf:///ospool/ap40/data/thomas.burton/yarp-containers/v1/yarp-egat-20260615-v1.sif
    osdf:///ospool/ap40/data/thomas.burton/yarp-containers/v1/yarp-crest-20260615-v1.sif
    osdf:///ospool/ap40/data/thomas.burton/yarp-containers/v1/yarp-pysis-xtb-20260615-v1.sif
    osdf:///ospool/ap40/data/thomas.burton/yarp-containers/v1/yarp-orca-6.0.1-20260615-v1.sif

Do not hardcode these exact paths throughout the code. Make them configurable.

======================================================================
DESIRED `yarp-osg` CLI
======================================================================

A useful first CLI could support:

    yarp-osg init .
    yarp-osg plan .
    yarp-osg submit .
    yarp-osg status .
    yarp-osg harvest .
    yarp-osg retry .
    yarp-osg cleanup .
    yarp-osg advance .
    yarp-osg run .

Suggested semantics:

- `plan`
  - inspect pickle and STATUS.json
  - discover ready tasks
  - generate a human-readable plan
  - do not submit anything

- `submit`
  - generate manifests and submit ready jobs
  - record Condor IDs

- `status`
  - summarize ready, submitted, idle, running, held, complete, failed, retried, quarantined

- `harvest`
  - collect finished outputs
  - call existing YARP scrapers
  - update pickle and STATUS.json

- `retry`
  - retry eligible transient failures
  - respect max retry counts

- `cleanup`
  - remove disposable scratch
  - preserve essential result files and diagnostics

- `advance`
  - one controller cycle:
      harvest → update state → plan → submit

- `run`
  - repeated controller cycles until complete or no progress is possible

The first implementation may be smaller, but the architecture should allow this.

======================================================================
BATCH SUBMISSION DESIGN
======================================================================

For thousands of jobs, prefer one submit description per task type and wave.

Example concept:

    crest_jobs.tsv
    pysis_gsm_jobs.tsv
    xtb_rpopt_jobs.tsv
    xtb_tsopt_jobs.tsv
    xtb_irc_jobs.tsv
    orca_rpopt_jobs.tsv
    orca_tsopt_jobs.tsv
    orca_irc_jobs.tsv

Each row should contain enough information to locate:

- task ID
- reaction ID/hash
- local task directory
- input filename(s)
- expected output filename(s)
- resource class
- retry count

Then use something like:

    queue task_id,task_dir,input_name from crest_jobs.tsv

or `initialdir = $(task_dir)` where appropriate.

Prefer one Condor cluster with many ProcIds over thousands of single-job clusters.

Record:

    ClusterId.ProcId

for every logical task.

A robust mapping should exist between:

    YARP task identity
    task directory
    manifest row
    ClusterId.ProcId
    attempt number

Use a durable machine-readable state file, not only logs.

SQLite, JSONL, or a structured JSON/YAML state file are acceptable. Prefer something concurrency-safe and easy to audit.

WORKER SCRIPT REQUIREMENTS
======================================================================

Generated worker scripts must:

- use `set -uo pipefail` or equivalent diagnostic-safe behavior
- use worker-local scratch
- print:
    hostname
    working directory
    task ID
    selected container
    key environment variables
- never reference submit-host `/home/...` paths
- invoke the task executable directly inside the HTCondor-selected container
- write expected outputs into worker scratch
- preserve meaningful stdout/stderr
- create a compact task result/status record
- avoid causing output-transfer holds merely because an optional output is absent
- return explicit exit codes
- distinguish:
    infrastructure failure
    container failure
    chemistry failure
    validation failure
    missing output
    timeout
    preemption/eviction

Where possible, use a standard result file such as:

    task_result.json

with fields like:

    task_id
    reaction_id
    task_type
    attempt
    exit_code
    status
    hostname
    start_time
    end_time
    runtime_seconds
    expected_outputs
    present_outputs
    error_category
    error_message

======================================================================
RETRY AND QUARANTINE
======================================================================

Support bounded retry behavior.

Suggested categories:

Transient infrastructure errors:
- retry automatically
- examples:
    transfer failure
    worker eviction
    container fetch failure
    site error
    temporary filesystem failure

Chemistry failure:
- retry only if configured
- examples:
    SCF convergence failure
    optimizer failure
    GSM failure
    missing TS
    invalid IRC

Configuration/programming failure:
- do not retry automatically
- examples:
    executable missing
    malformed input
    unsupported config
    container command not found

Use configurable values such as:

    max_infrastructure_retries: 3
    max_chemistry_retries: 1
    quarantine_after: 3

Quarantined tasks must remain inspectable.

Do not silently discard failures.

======================================================================
OUTPUT RETENTION
======================================================================

Do not return all worker scratch files.

Define per-task retention rules.

Always keep:

- final geometry
- essential energy/result summary
- task_result.json
- compact stdout/stderr
- files required by downstream YARP scraping
- files required for restart, only when justified

Usually discard:

- redundant temporary files
- large trajectories
- duplicate intermediates
- transient optimizer scratch
- unnecessary wavefunction files
- verbose logs after successful scraping, if configured

Preserve more files for failed tasks than successful tasks.

Make cleanup policy configurable.
