YARP SMILES parser refactor notes (planning file)
===============================================

Scope
-----
This file documents implementation logic for:
- atom-map persistence via yarpecule._atom_info
- yarp-native mapped smiles output using preserved atom maps
- stereo preservation in metadata but stripped from output smiles
- aromatic parser stabilization for lowercase/fused aromatic inputs
- persistence of atom metadata across reorder/separate/join operations

Atom info design
----------------
Use a dict keyed by current atom index:

atom_info[i] = {
  "index": int,                   # map number (from smiles or generated)
  "element": str,                 # lowercase element
  "isotope": int|None,
  "formal_charge": int,
  "explicit_hydrogens": int|None,
  "steric_marking": {
      "atom": "@"|"@@"|None,
      "bond": {neighbor_idx: "/"|"\\"}
  },
  "atomic_weight": float,
  "aromatic": bool,
}

Implementation notes
--------------------
1. smiles2adjmat()
   - Parse map index and stereo from bracket tokens.
   - Capture bond stereo tokens (/ and \\) between atom tokens.
   - Build atom_info entries for all parsed atoms, then add H entries in add_hydrogens().
   - Ensure hydrogens also receive map indices (generated if absent).
   - Reorder by map index if mappings are present.

2. xyz_from_smiles()
   - Return 5-tuple: (elements, geo, adj_mat, q, atom_info).
   - Keep old return shape only if backward compatibility switch is required.

3. yarpecule._read_structure()
   - SMILES path should assign self._atom_info from xyz_from_smiles().
   - Non-SMILES constructors should initialize _atom_info with generated indices and defaults.

4. yarpecule._order_atoms()
   - Any canonical/reordered index transform must also reorder _atom_info entries.
   - Preserve map numbers (index field) through reorder.

5. yarpecule.get_smiles()
   - Add writer toggle: writer="yarp" (default), writer="rdkit".
   - yarp writer uses atom_info index values directly for mapped output.
   - strip_stereo=True default for both canonical and mapped output strings.

6. join()/separate()
   - join: merge _atom_info from each parent and resolve map collisions deterministically.
   - separate: slice _atom_info with fragment indices without regenerating map numbers.

Aromatic handling strategy
--------------------------
Observed risk: current aromatic assignment relies on ring detection with remove_fused=True,
which can skip fused aromatic systems.

Preferred update:
- Use aromatic atom labels (lowercase) to define aromatic subgraph.
- Assign double bonds in aromatic subgraph with deterministic matching and valence checks.
- Keep fallback path for ring-local assignment if matching under-fills.

Warning suppression (noise control)
-----------------------------------
- RDKit: set RDLogger to ERROR around MolFromMolFile/MolToSmiles/Embed calls.
  This suppresses warnings/info but leaves errors.
- OpenBabel: SetOutputLevel(obError) around pybel.readfile/mol.write calls.
  This suppresses warnings/info/debug but keeps errors.
- Always restore previous logger level in finally blocks.

Failure modes to test
---------------------
1. Duplicate atom maps in input.
2. Mixed mapped/unmapped atoms in one smiles.
3. Ring closure labels above 9 (%10 syntax).
4. Fused aromatics (naphthalene/anthracene).
5. Aromatic heteroatoms with charges ([nH], [o+], etc.).
6. Stereo tags present in input and persisted but stripped from output.
7. join() map collisions between two inputs.
8. separate() preserving map IDs and metadata exactly.

Unit test plan summary
----------------------
- smiles parser tests:
  - map retention for heavy + explicit hydrogen maps
  - generated maps when absent
  - stereo extraction and storage
  - fused aromatic parsing and sane valence
- yarpecule tests:
  - _atom_info exists for all atoms after smiles init
  - get_smiles(writer="yarp") returns mapped smiles with preserved IDs
  - strip_stereo default removes @,/,\\ in outputs only
  - writer="rdkit" still available for fallback
  - join/separate retain _atom_info and map continuity

Open questions
--------------
1. Should map indexing be normalized to 1-based if input contains 0-based maps?
2. Should join() preserve conflicting map IDs or force unique IDs globally?
3. For canonical smiles, keep temporary RDKit path or move fully to yarp writer now?
